# astrbot_plugin_forward_merged_reply

该项目由 Codex 编写。

一个用于 AstrBot 的插件，让 Bot 能对“被引用的 QQ 合并转发消息”做出正常回应，而不是只能看到用户当前那一句话。

## 适用场景

插件只处理这一类消息：

- 用户发送了一条消息
- 这条消息里显式 `@bot`
- 同时引用了一条已有消息
- 被引用消息本身包含 QQ 合并转发内容

普通 `@bot` 对话不会进入深度解析流程。

## 行为说明

当命中条件时，插件会做两件事：

1. 先发一条简短提示，例如：`让我康康`
2. 递归展开被引用消息中的合并转发内容，并由插件主动调用当前会话模型生成回复

模型最终如何回答，不被强制限定为“总结”：

- 该回应就回应
- 该总结就总结
- 该提取信息就提取信息

也就是说，这个插件做的是“补上下文”，不是“替你规定回答任务”。

## 触发方式

插件不会等 AstrBot 先进入默认 LLM 流程，而是直接在普通消息事件阶段判断：

- 如果只是普通 `@bot`，直接跳过
- 如果是“引用消息 + @bot + 被引用内容本身含合并转发”，插件会接管这条消息
- 先发送短提示，再自己调用当前会话使用的模型回复

这样即使用户发送的是“只有引用和 @，没有正文”的消息，也能正常触发。

## 支持范围

- 被引用的单个合并转发
- 并列多个合并转发
- 合并转发里再次嵌套合并转发
- 私聊和群聊中的“引用 + @bot”场景

## 处理策略

为了避免每条 `@bot` 都被重解析，插件采用强短路逻辑：

1. 先检查插件是否启用
2. 再检查当前消息里是否显式 `@bot`
3. 再检查是否存在被引用消息
4. 最后只检查“被引用的内容本身”里是否真的含有合并转发 `Forward` 结构

只有全部满足，才会进入展开逻辑。

这意味着：

- 引用普通图片、普通文本、普通回复，不会触发
- 当前消息其他位置就算带了转发结构，也不会因为旁路数据误触发
- 是否触发由代码硬判断，不依赖 LLM 自行理解

## 配置项

- `enabled`
  控制插件总开关。

- `processing_message`
  正式回复前先发送的短提示。

- `system_guidance`
  注入给模型的说明文字。默认会强调“不要把任务自动改写成纯总结”。

- `reply_max_chars`
  最终回复的目标最大字数。插件会在 LLM 返回后尽量压缩到这个长度内。

- `max_depth`
  最大递归展开层数，用于限制嵌套深度。

- `max_nodes`
  单次最多展开的节点数量，避免特别长的记录拖慢模型。

- `max_chars`
  注入给模型的最大字符数，超出则自动截断。

## 文件说明

- `main.py`：插件入口和 AstrBot hook 编排
- `fmr_forward.py`：合并转发识别、抓取、展开、渲染
- `fmr_persona.py`：当前会话人格解析与缓存
- `fmr_reply.py`：回复压缩处理
- `__init__.py`：包标记，保证包内相对导入稳定
- `metadata.yaml`：插件元数据
- `_conf_schema.json`：配置定义
- `requirements.txt`：依赖声明
- `LICENSE`：MIT 协议

## 兼容性说明

由于不同 QQ 适配层对“引用消息”“合并转发节点”的字段命名可能不同，插件内部使用的是较保守的“属性探测式”兼容策略：

- 会尝试多个常见字段名
- 优先提取纯文本内容
- 对嵌套结构做深度和长度限制
- 人格注入优先读取当前会话人格，找不到时再回退到默认人格/当前 provider 人格

如果你所用适配器的字段命名与常见实现差异较大，可能还需要按实际平台消息结构再补一次适配。
